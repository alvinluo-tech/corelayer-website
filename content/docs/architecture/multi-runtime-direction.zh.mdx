当前的单 TypeScript 守护进程是一个过渡阶段。长期目标是由 Tauri/Rust 核心监督的多运行时架构。

## 当前架构

```text
Tauri 桌面应用
  └─ 单一 Node.js 守护进程
     └─ 所有服务在一个进程中
```

这在当前范围内运作良好，但存在局限性：

- 单点故障
- 所有服务共享同一进程
- TypeScript 对某些工作负载存在固有的性能限制

## 目标架构

```text
Tauri/Rust 核心
  ├─ 代理运行时（TypeScript）
  ├─ 工具运行时（TypeScript）
  ├─ 编码运行时（TypeScript）
  ├─ 语音运行时（TypeScript）
  ├─ 记忆运行时（TypeScript）
  ├─ 调度运行时（TypeScript）
  └─ 计算机控制运行时（Rust）
```

每个运行时：

- 作为受管理的子进程运行
- 通过运行时协议通信
- 可以独立重启
- 有自己的资源限制

## 运行时协议

`@jarvis/runtime-protocol` 包定义了 Tauri 核心与受管理运行时之间的通信契约：

- **生命周期** — 启动、停止、重启、健康检查
- **消息传递** — 类型化的请求/响应
- **事件** — 运行时到核心的通知
- **资源限制** — 内存、CPU、时间约束

## 已就绪的包

| 包 | 状态 | 用途 |
|----|------|------|
| `@jarvis/runtime-core` | 活跃 | 运行时管理核心 |
| `@jarvis/runtime-protocol` | 活跃 | 协议定义 |
| `@jarvis/execution-environment` | 新建 | 执行抽象 |

## 迁移路径

迁移是渐进式的：

1. **阶段 1**（当前） — 带有包边界的单一守护进程
2. **阶段 2** — 将语音管线提取到独立运行时
3. **阶段 3** — 将工具执行提取到独立运行时
4. **阶段 4** — 完整的多运行时，由 Rust 核心监督

每个阶段都是向后兼容的。前端不会改变 — 它仍然与同一个 HTTP API 通信。

## 为什么需要多运行时？

- **可靠性** — 一个运行时的崩溃不会影响其他运行时
- **性能** — Rust 用于性能关键路径
- **可扩展性** — 运行时可以独立扩展
- **灵活性** — 根据需要混合使用 TypeScript 和 Rust
