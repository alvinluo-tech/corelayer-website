目前的單一 TypeScript 守護程式是一個過渡階段。長期目標是由 Tauri/Rust 核心監督的多執行階段架構。

## 目前架構

```text
Tauri 桌面應用程式
  └─ 單一 Node.js 守護程式
     └─ 所有服務在同一程序中
```

這在目前的範圍內運作良好，但有其限制：

- 單點故障
- 所有服務共享同一程序
- TypeScript 對某些工作負載有固有的效能限制

## 目標架構

```text
Tauri/Rust 核心
  ├─ 代理執行階段（TypeScript）
  ├─ 工具執行階段（TypeScript）
  ├─ 程式碼執行階段（TypeScript）
  ├─ 語音執行階段（TypeScript）
  ├─ 記憶執行階段（TypeScript）
  ├─ 排程器執行階段（TypeScript）
  └─ 電腦控制執行階段（Rust）
```

每個執行階段：

- 作為受管理的子程序運行
- 透過執行階段協定進行通訊
- 可以獨立重新啟動
- 有自己的資源限制

## 執行階段協定

`@jarvis/runtime-protocol` 套件定義了 Tauri 核心和受管理執行階段之間的通訊契約：

- **生命週期** — 啟動、停止、重新啟動、健康檢查
- **訊息傳遞** — 型別化請求/回應
- **事件** — 執行階段到核心的通知
- **資源限制** — 記憶體、CPU、時間約束

## 已就位的套件

| 套件 | 狀態 | 用途 |
|---------|--------|---------|
| `@jarvis/runtime-core` | 活躍 | 執行階段管理核心 |
| `@jarvis/runtime-protocol` | 活躍 | 協定定義 |
| `@jarvis/execution-environment` | 新增 | 執行抽象層 |

## 遷移路徑

遷移是漸進式的：

1. **階段 1**（目前）— 具有套件邊界的單一守護程式
2. **階段 2** — 將語音管線提取為獨立執行階段
3. **階段 3** — 將工具執行提取為獨立執行階段
4. **階段 4** — 完整的多執行階段，由 Rust 核心監督

每個階段都向後相容。前端不會改變 — 它仍然與相同的 HTTP API 通訊。

## 為什麼要多執行階段？

- **可靠性** — 一個執行階段的崩潰不會影響其他執行階段
- **效能** — 對效能關鍵路徑使用 Rust
- **可擴展性** — 執行階段可以獨立擴展
- **靈活性** — 根據需要混合 TypeScript 和 Rust
