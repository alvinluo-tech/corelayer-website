現在の単一TypeScriptデーモンは移行段階です。長期目標は、Tauri/Rustコアによって監督されるマルチランタイムアーキテクチャです。

## 現在のアーキテクチャ

```text
Tauriデスクトップアプリ
  └─ 単一Node.jsデーモン
     └─ すべてのサービスが単一プロセス内
```

これは現在のスケールではうまく機能しますが、いくつかの制限があります:

- 単一障害点
- すべてのサービスが同じプロセスを共有
- TypeScriptには一部のワークロードに対する固有のパフォーマンス限界がある

## 目標アーキテクチャ

```text
Tauri/Rustコア
  ├─ Agentランタイム（TypeScript）
  ├─ Toolランタイム（TypeScript）
  ├─ Codingランタイム（TypeScript）
  ├─ Voiceランタイム（TypeScript）
  ├─ Memoryランタイム（TypeScript）
  ├─ Schedulerランタイム（TypeScript）
  └─ Computer Controlランタイム（Rust）
```

各ランタイム:

- 管理されたサブプロセスとして実行
- ランタイムプロトコルを通じて通信
- 独立して再起動可能
- 独自のリソース制限を持つ

## ランタイムプロトコル

`@jarvis/runtime-protocol`パッケージは、Tauriコアと管理ランタイム間の通信契約を定義します:

- **ライフサイクル** — 開始、停止、再起動、ヘルスチェック
- **メッセージング** — 型付きリクエスト/レスポンス
- **イベント** — ランタイムからコアへの通知
- **リソース制限** — メモリ、CPU、時間制約

## 既に導入済みのパッケージ

| パッケージ | ステータス | 用途 |
|---------|--------|---------|
| `@jarvis/runtime-core` | アクティブ | ランタイム管理コア |
| `@jarvis/runtime-protocol` | アクティブ | プロトコル定義 |
| `@jarvis/execution-environment` | 新規 | 実行抽象化 |

## 移行パス

移行は段階的に行われます:

1. **フェーズ1**（現在） — パッケージ境界を持つ単一デーモン
2. **フェーズ2** — 音声パイプラインを独自のランタイムに分離
3. **フェーズ3** — ツール実行を独自のランタイムに分離
4. **フェーズ4** — Rustコア監督による完全なマルチランタイム

各フェーズは後方互換性があります。フロントエンドは変更されません — 同じHTTP APIを使用し続けます。

## なぜマルチランタイムなのか？

- **信頼性** — あるランタイムのクラッシュが他のランタイムに影響しない
- **パフォーマンス** — パフォーマンスクリティカルなパスにRustを使用
- **スケーラビリティ** — ランタイムを独立してスケール可能
- **柔軟性** — 適切にTypeScriptとRustを混在
