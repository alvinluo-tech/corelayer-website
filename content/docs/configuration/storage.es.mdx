CoreLayer admite tres backends de almacenamiento que pueden intercambiarse en caliente durante la ejecucion.

## Modos de Almacenamiento

| Modo | Descripcion | Caso de Uso |
|------|-------------|-------------|
| **SQLite** | Base de datos de archivo local (por defecto) | Uso personal, local primero |
| **Supabase** | PostgreSQL en la nube con sincronizacion en tiempo real | Sincronizacion multi-dispositivo |
| **PostgreSQL** | PostgreSQL autoalojado | Uso en equipo/empresa |

## SQLite (Por Defecto)

SQLite es el modo de almacenamiento por defecto. Los datos se almacenan localmente en un archivo `.db`:

```json
{
  "storage": {
    "mode": "sqlite",
    "path": "~/.corelayer/data.db"
  }
}
```

No requiere dependencias externas. Funciona completamente sin conexion.

## Supabase

Conecta a Supabase para sincronizacion en la nube:

```json
{
  "storage": {
    "mode": "supabase",
    "url": "https://your-project.supabase.co",
    "anonKey": "eyJ..."
  }
}
```

Configura Supabase:
1. Crea un proyecto en [supabase.com](https://supabase.com)
2. Ejecuta los scripts de migracion de CoreLayer
3. Aniade la URL de tu proyecto y la clave anon

## PostgreSQL

Conecta a una instancia de PostgreSQL autoalojada:

```json
{
  "storage": {
    "mode": "postgres",
    "connectionString": "postgresql://user:pass@localhost:5432/corelayer"
  }
}
```

## Cambio en Caliente

Puedes cambiar los modos de almacenamiento en tiempo real desde el Centro de Control:

1. Abre **Ajustes → Almacenamiento**
2. Selecciona el nuevo modo
3. Introduce los detalles de conexion
4. Haz clic en **Cambiar**

CoreLayer migrara los datos entre modos automaticamente.

## Esquema de Datos (Data Schema)

CoreLayer utiliza un esquema de base de datos local-first altamente estructurado, que almacena las siguientes entidades principales:

*   **workspaces** — espacios de trabajo de nivel superior que gestionan objetivos, desglose de tareas y estado de ejecución
*   **projects** — carpetas de proyectos específicas dentro de los espacios de trabajo, incluidos specs, stacks tecnológicos y rutas raíz locales
*   **tasks** — nodos de tareas que forman el gráfico de dependencias para la ejecución
*   **artifacts** — entregables finales valiosos, separando los archivos/informes del ruido de la terminal y los logs
*   **agent_runs** — sesiones de ejecución individuales de agentes, rastreando estados, latencia, enrutamiento y registros
*   **approval_requests** — acciones de alto riesgo pendientes que esperan la decisión de aprobación manual del usuario
*   **permission_memories** — almacenamiento en caché temporal de decisiones para evitar ventanas emergentes de permisos repetitivas
*   **conversations** / **messages** — historiales de conversaciones de chat y registros de mensajes individuales
*   **tool_call_logs** — registros de auditoría que capturan duración, argumentos, niveles de riesgo y resultados
*   **app_connections** — servidores MCP locales o remotos configurados y conexiones de aplicaciones
*   **model_profiles** — configuraciones de gateway de modelos y endpoints de proveedores
*   **memories** — recuerdos a largo plazo de Jarvis (hechos, preferencias, contexto, fijados)
*   **scheduled_tasks** — cron jobs que activan tareas automatizadas y habilidades
*   **environment_sessions** / **environment_events** — contenedores de sandbox aislados y flujos de eventos
*   **articles** — entradas de listas de lectura
*   **reviews** — resúmenes de revisión diarios/semanales
